# OA流程智能审核平台建设方案

## 1. 项目概述与设计思路
本系统旨在构建一个专门针对OA系统的智能审核平台（SaaS/多租户模式）。通过“流程信息配置 + AI大模型审核”的核心机制，实现对OA审批流程的个性化、自动化辅助审核。

系统在逻辑上划分为 **前台业务端**（面向普通用户/审批者）和 **后台管理端**（面向租户管理员及系统管理员）。

## 2. 功能架构设计

### 2.1 前台业务端（用户/审批工作台）
主要面向具体组织的租户成员。系统采用**动态菜单权限机制**，不同角色的用户仅能看到其有权访问的模块。核心分为三大板块：

#### 1. 智能待办审核（核心工作台）
这是用户日常最高频使用的模块，强调**审核逻辑透明化**与**按需定制**：
*   **审核透明化**：
    *   **规则清单可视化**：在审批界面侧边栏，清晰列出当前流程加载的所有审核点。
    *   **命中与推理展示**：直观展示通过/不通过的规则，点击可展开查看AI的推理依据（引用制度原文或具体计算过程）。
*   **个性化偏好定制（用户级）**：
    *   **规则开关**：针对管理员设定为“可选”的通用规则，用户可自主决定开启或关闭。
    *   **私有规则叠加**：用户可添加仅对自己可见的“私有关注点”（自定义Prompt），例如“帮我检查合同金额是否大写”。
    *   **力度调整**：在基准范围内调整AI敏感度（如调整为“严格模式”）。
*   **操作闭环**：
    *   输出审核建议（通过/驳回/修改建议）及一键跳转OA处理。

#### 2. 定时任务中心
针对不习惯实时处理或需要批量处理的场景：
*   **任务配置**：用户可建立个人Cron任务（如“每天下午4点”、“每周五”）。
*   **日报/周报推送**：系统汇总指定时间段内的待办流程及风险点，通过邮件/IM推送摘要。
*   **历史记录查看**：查看过往的推送记录及当时的审核快照。

#### 3. 归档流程复盘
针对已完结数据的后评估与查询：
*   **合规性复盘**：对已结束的流程进行“事后诸葛亮”式的AI复核，分析整体合规率。
*   **全量历史检索**：支持按时间、部门、流程类型检索所有的AI审核记录（包括当时的Prompt、AI回复、用户采纳情况）。

### 2.2 后台管理端
分为两级管理权限，重点在于资源配置与规则定义。

#### A. 租户管理员配置（组织级）
*   **知识库与规则配置（模块化可选）**：
    *   改变以往“规则优先、制度兜底”的硬逻辑，改为**灵活勾选模式**。针对某个流程，管理员可配置：
        *   **模式A（仅规则库）**：仅使用结构化的Checklist（如“金额<5000”），精准度高，适合强逻辑校验。
        *   **模式B（仅制度库/RAG）**：仅挂载PDF/Word制度文档，让AI基于文档回答，适合非结构化内容审查。
        *   **模式C（混合模式）**：同时启用，双重校验。
*   **审核规则分级管理**：
    *   **强制执行（Mandatory）**：红线规则，用户不可见或不可改。
    *   **默认开启/关闭**：给予用户自定义权限的基础规则。
*   **全链路记录管理**：
    *   **日志留存设置**：配置AI推理日志、用户修改偏好记录的保留时长（满足审计合规要求）。

#### B. 系统管理员配置（平台级）
*   **租户与资源管理**：管理租户开通、Token配额限制、并发数控制。
*   **OA集成底座**：维护多版本OA数据库适配脚本。
*   **全局监控**：系统健康度、模型响应时间、API调用成功率分析。

## 3. 关键可配置项详情

| 模块 | 配置项说明 |
| :--- | :--- |
| **OA平台集成** | 支持选择OA类型（首发泛微E9），配置JDBC连接及数据库表结构映射。 |
| **知识库策略** | **核心配置**：针对特定流程，可勾选【启用规则库】、【启用制度库(RAG)】或【两者并行】。 |
| **流程选择** | 支持按目录、具体路径或ID选择需要AI介入的流程。 |
| **规则属性** | 定义规则是【强制/默认开启/默认关闭】，决定用户端的修改权限。 |
| **历史记录** | 配置审核快照、AI推理过程、用户操作日志的保留策略（如：永久保存/保存3年）。 |
| **AI配置** | 模型选型（本地/云端）、Prompt模板、上下文窗口大小限制。 |
| **配置优先级** | **层级逻辑**：租户强制规则 > 用户私有规则 > 租户默认规则。 |

## 4. 技术架构方案

### 4.1 技术选型（Go + Python 混合架构）
采用 **Go处理高并发业务** + **Python处理AI智能服务** 的双引擎模式。

*   **前端**：
    *   **Nuxt 3 + TypeScript**，配合 **Ant Design Vue**，支持服务端渲染（SSR）。
    *   实现**动态路由**（基于RBAC模型），根据用户权限加载“待办”、“定时”、“归档”菜单。
*   **后端服务（Go语言）**：
    *   *定位*：业务中台。
    *   *职责*：用户鉴权、动态菜单下发、OA数据轮询、任务调度（Cron）、历史记录持久化（Log Storage）。
*   **AI服务（Python语言）**：
    *   *定位*：智能引擎。
    *   *职责*：LLM交互、RAG检索（向量库查询）、OCR解析。
*   **数据库与存储**：
    *   **PostgreSQL**：业务数据 + 向量数据。
    *   **MongoDB/Elasticsearch**（新增）：专门用于存储海量的审核历史记录、AI对话原文、推理过程快照，便于后期审计和复盘检索。

### 4.2 核心问题与解决方案
1.  **用户自定义与统一管控的冲突**：
    *   *解决方案*：数据库引入`rule_scope`和`is_locked`字段。前端渲染时，强制规则锁定展示，可选规则开放Toggle开关。用户自定义的Prompt作为额外List追加到请求中。
2.  **知识库的灵活组合**：
    *   *解决方案*：在流程配置表中增加 `kb_mode` 字段（枚举值：Rules_Only, RAG_Only, Hybrid）。Python端根据该字段决定是仅调用LangChain的Checklist Chain，还是Retrieval Chain，亦或是并行执行后合并结果。
3.  **历史记录的可追溯性**：
    *   *解决方案*：每次审核结束，Go服务将“输入（OA表单）+ 配置（生效规则）+ 过程（AI思维链）+ 结果（建议）+ 用户反馈（是否采纳）”打包存入日志库（ES/Mongo），确保任何一次审核都有据可查。

## 5. 非功能性需求
*   **数据完整性**：必须保证所有的审核记录不可篡改，支持审计导出。
*   **安全性**：敏感字段（薪资、身份证）在Go层进行正则脱敏后，再发送给Python层处理。
*   **可观测性**：链路追踪，记录请求在Go和Python间的流转。
