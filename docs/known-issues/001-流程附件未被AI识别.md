# 001 - 流程附件未被 AI 识别

## 基本信息

- 缺陷编号：`001`
- 标题：流程附件未被 AI 识别
- 状态：待处理
- 优先级：高
- 影响范围：审核工作台、归档复盘、审核结论可靠性
- 首次确认时间：2026-05-01

## 现象描述

当前审核链路中，AI 仅接收到主表/明细表字段文本和审批流文本，未接收到流程附件的文件信息或内容。  
这会导致涉及合同、发票、报价单、扫描件等“关键证据在附件内”的流程被误判为低风险，或给出不完整结论。

## 代码阅读结论（现状）

### 1) OA 数据提取层未产出附件结构

- `go-service/internal/pkg/oa/adapter.go` 中 `ProcessData` 结构仅包含：
  - `MainData`
  - `DetailTables`
- 适配器接口 `FetchProcessData` 仅返回上述结构，没有附件字段。

### 2) E9 适配器仅识别“字段类型=file”，未拉取附件实体

- `go-service/internal/pkg/oa/ecology9.go` 的 `mapFieldType()` 将 `fieldhtmltype=6` 映射为 `file`。
- 但 `FetchProcessData()` 只查询主表和明细表，不查询附件关联表，不构建附件元数据/下载地址/文本内容。

### 3) Prompt 组装层没有附件占位与注入

- `go-service/internal/service/audit_prompt_builder.go` 仅注入：
  - `{{main_table}}`
  - `{{detail_tables}}`
  - `{{rules}}`
  - `{{current_node}}`
  - `{{flow_history}}`
  - `{{flow_graph}}`
- `go-service/internal/service/archive_prompt_builder.go` 同样仅注入主表/明细表/流程文本，不包含附件变量。

### 4) AI 调用协议为纯文本消息

- `go-service/internal/pkg/ai/caller.go` 中 `ChatRequest` 只有 `SystemPrompt` 与 `UserPrompt` 文本字段。
- `go-service/internal/pkg/ai/openai_compat.go` 将请求固定组装为两条文本 message（system + user），无附件对象或多模态输入结构。

## 根因分析

1. **数据模型缺口**：`ProcessData` 未定义附件维度。  
2. **适配器缺口**：OA 适配层没有统一“附件提取协议”。  
3. **提示词缺口**：Prompt 模板链路没有附件占位和序列化策略。  
4. **调用能力缺口**：AI 请求协议仅支持文本，缺少附件/多模态扩展位。

## 风险与业务影响

- 合规漏检：关键证据在附件内时，审核结论可能偏乐观。
- 结果不可解释：审核结论无法引用附件证据，降低可追溯性。
- 人工复核压力上升：系统结论与人工判断偏差变大。

## 建议修复方案（MVP）

### 阶段 A：先实现“附件元数据入模（不解析文件正文）”

- 目标：让 AI 至少“知道有附件、是什么附件、多少个附件”。
- 建议动作：
  - 扩展 `ProcessData`，新增 `Attachments` 字段（数组）。
  - 在 OA 适配器中补齐附件查询与标准化映射（名称、类型、大小、上传人、时间、引用字段）。
  - 在审核/归档 Prompt 中新增 `{{attachments}}` 占位并注入结构化文本。
- 验收标准：
  - AI 输入中可见附件列表；
  - 审核日志可追踪附件摘要；
  - 不改动现有非附件流程行为。

### 阶段 B：支持“可解析附件正文（按类型分批）”

- 优先类型：`pdf`、`docx`、`xlsx`、`txt`、图片 OCR。
- 建议动作：
  - 新增附件解析服务，抽取文本摘要与关键字段；
  - 解析结果写入 `Attachments[].ExtractedText/Highlights`；
  - Prompt 仅注入截断后的高价值文本，控制 token 成本。
- 验收标准：
  - 指定样例流程中，AI 能引用附件内容进行判定；
  - 单次审核 token 增幅可控；
  - 解析失败时有降级策略（仅元数据）。

### 阶段 C：策略与治理

- 附件白名单/黑名单（类型、大小、数量）。
- 安全策略（脱敏、恶意文件防护、临时文件清理）。
- 审计策略（记录附件摘要来源与解析状态）。

## 待办拆解

- [ ] 定义统一附件 DTO 与 `ProcessData` 扩展结构。
- [ ] 在 E9 适配器实现附件查询并回填 DTO。
- [ ] 审核与归档 Prompt 增加 `{{attachments}}` 注入逻辑。
- [ ] 增加单元测试：无附件/单附件/多附件/异常附件。
- [ ] 增加集成测试：附件参与审核时的结论稳定性。
